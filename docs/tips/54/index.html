Tip #54 intermediate Act 3

使用 /btw 侧链对话

在 Claude 工作时并行提问，不中断主任务

⏱ 32:00 📁 power-user
标签： btwside-conversationoverlayworkflow

描述

/btw (By The Way) 命令开启侧链对话模式，让你在 Claude 执行主任务时并行提问，获得即时回答而不中断工作流程。

为什么使用 /btw

  • 不中断流程 - 主任务继续执行，侧链问题并行处理
  • 即时回答 - 答案显示在可关闭的浮层中
  • 保持专注 - 不会打乱主对话的历史记录
  • 只读模式 - 侧链对话不调用工具，不影响项目状态

使用方式

基本用法

当 Claude 正在处理任务时（显示思考或执行工具），随���输入：

/btw 这个函数是做什么的？

工作原理

主对话：正在执行复杂任务...
    │
    ├─── 思考中...
    ├─── 读取文件...
    └─── 运行测试...

侧链：/btw 这个变量是什么意思？
      │
      └── 即时浮层回答（不影响主任务）

关闭浮层

答案显示后，可以：

  • 点击关闭按钮
  • 按 Esc 键
  • 点击浮层外部区域

适用场景

1. 代码理解

# Claude 正在重构代码时
/btw useEffect 的依赖数组怎么做？

2. 快速查证

# Claude 正在编写测试时
/btw JWT token 默认有效期是多少？

3. 相关疑问

# Claude 正在修复 bug 时
/btw 这个错误码 401 是什么意思？

4. 临时想法

# Claude 正在实现功能时
/btw 我们是不是应该先加个缓存层？

与主对话的区别

特性主对话/btw 侧链
工具调用✅ 读写文件、运行命令❌ 只读模式
历史记录✅ 保存到对话历史❌ 不记录
中断任务⚠️ 会打断当前执行✅ 不影响主任务
显示方式内联在对话中浮层覆盖显示
上下文共享完整上下文独立简短上下文

最佳实践

1. 用于简单问题

# ✅ 适合 - 简单、独立的问题
/btw 这个 CSS 属性是做什么的？
/btw TypeScript 的 `unknown` 和 `never` 有什么区别？

# ❌ 不适合 - 需要工具调用的复杂任务
/btw 帮我重写这个组件  # 应该在主对话中处理

2. 保持问题简洁

# ✅ 好 - 简洁直接
/btw 这个 API 的返回格式？

# ⚠️ 可行但不是最优 - 问题太长
/btw 我们之前讨论的那个用户认证流程，具体的 token 存储位置是在 localStorage 还是 cookie？我忘了

3. 避免重复提问

# 如果主对话中已回答过类似问题，优先查看历史
# 而不是每次都用 /btw 重复询问

4. 合理使用时机

# ✅ 好的时机
- Claude 正在执行长时间操作（安装依赖、运行测试）
- Claude 正在思考复杂方案
- 你突然想到一个相关问题

# ⚠️ 考虑主对话
- 需要查看文件内容
- 需要修改代码
- 问题与主任务密切相关

注意事项

  1. 只读限制 - /btw 不能读取项目文件或执行命令
  2. 上下文有限 - 侧链对话上下文比主对话更短
  3. 不被记录 - 侧链问题和答案不会保存到对话历史
  4. 手动记录 - 重要答案需要在主对话中确认或记录

实战示例

场景：重构中的疑问

# 主对话
用户：帮我重构 AuthContext，使用更好的类型定义
Claude：好的，让我先查看当前实现...
      [正在读取 AuthContext.tsx...]
      [正在分析类型定义...]

# 侧链提问（不中断重构）
/btw Context 的 displayName 有什么用？

# 浮层即时回答
displayName 用于 React DevTools 中显示组件名称，
方便调试。不影响运行时行为。

场景：编写测试时的疑问

# 主对话
用户：为用户认证模块编写单元测试
Claude：我来编写测试...[创建测试文件...]

# 侧链提问
/btw beforeEach 和 afterEach 的执行顺序？

# 浮层即时回答
beforeEach → test → afterEach → test → afterEach → ...
每个测试前执行 beforeEach，测试后执行 afterEach
